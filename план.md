Техническая спецификация

Школьная библиотечная система для Каракалпакстана. Production-grade перепроектирование с нуля.

---

## 1. Технологический стек

| Слой | Технология |
|---|---|
| Backend | Python 3.13, Django 5.x |
| API | Django REST Framework, JWT (simplejwt), generics, ViewSets |
| База данных | PostgreSQL (prod) / SQLite (dev) |
| Кэш/очереди | Redis + Celery (периодические задачи) |
| Frontend | Django Templates + HTMX 2.x (polling для near-realtime), адаптивная вёрстка (desktop/tablet/mobile) |
| Хранилище файлов | Облако (S3-совместимое) — обложки книг, со сжатием |
| QR | html5-qrcode + HMAC-токены (TTL 2 мин) |
| Безопасность | django-axes (5 попыток → бан 1 час), CSRF, роль-based permissions |
| i18n | 4 языка: uzbek, русский, каракалпакский, английский |
| ИИ (ограниченно) | Внешний бесплатный API (напр. Gemini) — **только** для генерации еженедельных челленджей, разово в фоне |
| Тема | Светлая/тёмная (тёмная — контрастная, близкая к чёрной, не смягчённая) |

## 2. Архитектура приложений (Django apps)

Разделение по доменам, НЕ по ролям:

```
apps/
├── accounts/        # User, роли, JWT, сброс паролей
├── schools/         # District, School, Class, переводы учеников/учителей
├── catalog/         # Textbook, TextbookStock, RegularBook, Category
├── loans/           # TextbookLoan, RegularBookLoan, QR/HMAC сервис, "корзина"
├── gamification/    # XP, Level, Achievement, Streak, Challenge (недельные тесты)
├── notifications/   # News (2 уровня видимости), push, in-app
├── stats/           # Логи действий, инвентаризация, лидерборды
└── core/            # Общие миксины, SchoolScopedModel, утилиты, локали

api/v1/              # DRF-слой: serializers, permissions, viewsets (тонкая обёртка над services)
dashboard/           # ЕДИНЫЙ UI-слой для всех ролей (не разделять на 3 приложения!)
```

**Ключевой архитектурный принцип:** вся бизнес-логика — в `services.py` внутри соответствующего домена. Views (Django) и ViewSets (DRF) — только тонкие обёртки, вызывающие сервисы. Это исключает дублирование логики между HTMX-интерфейсом и API.

## 3. Роли и права доступа

| Роль | Возможности |
|---|---|
| **Суперадмин** | Создаёт районы → школы (автосоздание школьного админа) → видит рейтинг школ (по среднему XP/книгам на ученика, не по абсолютным числам) → сбрасывает пароли школьным админам |
| **Школьный админ (библиотекарь)** | Управляет учениками/учителями/классами/книгами/учебниками своей школы; выдача/сдача через QR; ручной аварийный возврат; сброс паролей ученикам/учителям; экспорт Excel; публикует школьные новости; инвентаризация |
| **Учитель-предметник** | Видит/берёт книги и учебники своего предмета (дефолтный фильтр, можно снять при замене коллеги); видит общешкольный лидерборд; не участвует в геймификации |
| **Классный руководитель** (доп. роль поверх учителя, many-to-many к классам) | Read-only: видит своих учеников (книги, учебники, статус), рейтинг своего класса. Не может ничего выдавать/сбрасывать |
| **Ученик (5–11 класс)** | Личный кабинет, поиск/бронирование обычных книг через QR, геймификация (XP, уровни, достижения, стрики), недельные челленджи, лидерборд школы |
| **Ученик (1–4 класс)** | НЕТ аккаунта/входа. Существует только как запись (ФИО, класс) для учёта учебников, которые выдаёт/принимает вручную админ |

Мультитенантность реализуется через абстрактную модель `SchoolScopedModel` (поле `school`) + менеджер `SchoolScopedManager.for_user(user)`, который автоматически фильтрует по школе, кроме суперадмина.

## 4. Районы, школы, классы

### Районы и школы
- Суперадмин сначала создаёт **район** (справочник), затем создаёт **школу**, выбирая район из списка (не текстом).
- Уникальность школы — по паре **(название, район)**, проверка нечувствительна к регистру/пробелам.
- При создании школы автоматически создаётся школьный админ (логин/пароль генерируются).
- Пароль **нигде не отображается повторно** в интерфейсе — только один раз при создании/сбросе, через автоматически формируемый и скачиваемый Excel-файл (Школа | Район | ФИО админа | Логин | Пароль).

### Классы
- Сущность "Класс" = номер (1–11) + параллель (текстовое поле, не жёстко буквы) + **язык обучения** (рус/узб/каракалп/англ) + учебный год.
- К классу привязан комплект положенных учебников (список предмет→учебник).
- К классу могут быть прикреплены классные руководители (many-to-many).

### Автоперевод класса (1 сентября)
- Диапазон 1–11. После 11 — статус "выпускник"/архив (не удаление, просто флаг `status='graduated'`, история сохраняется).
- Защита от повторного выполнения: журнал переводов (school + academic_year + timestamp), проверка перед запуском.
- Операция атомарна (транзакция) — не должно быть "наполовину переведённой" школы.
- Переход 4→5 класс: ученик уже существовал как запись (см. ниже), при переходе в 5 админ вручную нажимает отдельную кнопку "активировать доступ 5 класса" — генерирует логины/пароли всем перешедшим в 5 класс разом, автоматически экспортирует Excel (ФИО | Класс | Логин | Пароль). Это НЕ привязано жёстко к моменту автоперевода — отдельное ручное действие.
- С момента активации доступа ученик автоматически включается в геймификацию, челленджи, школьный рейтинг (флаг "участвует в геймификации" ставится вместе с активацией доступа).

## 5. Пользователи 1–4 класс vs 5–11 класс

- **1–4 класс:** только запись в БД (ФИО, класс, школа) — нужна исключительно для учёта учебников. Нет логина, нет входа. Все операции с учебниками выполняет админ вручную через профиль ученика. Не участвуют ни в челленджах, ни в геймификации, ни в рейтингах. Фигурируют только в отчётности по учебникам (сколько выдано/в наличии).
- **5–11 класс:** полноценный User с логином/паролем, участвует во всём функционале.

## 6. Создание и управление пользователями

### Ученики/учителя — создание
- **Вручную** — форма (ФИО, класс), логин/пароль автогенерируются.
- **Массовый импорт (CSV/Excel)** — с кнопкой "Скачать шаблон" (колонки: ФИО | Класс). После загрузки — валидация (класс должен существовать или создаётся/ошибка по строке), система создаёт пользователей, автоматически экспортирует итоговый Excel с логинами/паролями.
- **ФИО, логины, пароли — строго на латинице**, даже для русскоязычных классов.
- У учителя дополнительно указывается **предмет преподавания**.

### Профиль (редактируется пользователем после входа)
- Ученик: ФИО, класс (readonly), логин (readonly), смена пароля.
- Учитель: то же + предмет.
- Фото профиля — отложено на будущее.

### Сброс пароля — строго иерархически
- Суперадмин → школьным админам.
- Школьный админ → ученикам/учителям своей школы (поиск → профиль → "сбросить пароль").
- Никто не сбрасывает себе или "вбок"/"вверх".
- При сбросе: логин не меняется, генерируется новый пароль, автоматически экспортируется Excel с обновлённым паролем, **все активные сессии пользователя инвалидируются**.
- Восстановление пароля через email/SMS не реализуется — полностью офлайн-процесс через админа.

## 7. Перевод учеников/учителей между школами

Двухфазный процесс:
1. **Уход** — админ старой школы подтверждает уход; обязательное условие — все книги и учебники сданы (проверка перед подтверждением).
2. **Приём** — админ новой школы принимает ученика; видит полную историю (книги, учебники, XP, баллы); выдаёт учебники заново (учебники не "переезжают" физически между школами — старая школа возвращает их в свой оборот при уходе, новая выдаёт свои).

Между уходом и приёмом — промежуточный статус (например, "ожидает перевода"), видимый обеим школам.

Ведётся лог переводов (кто, откуда, куда, когда, кто инициировал) — для отчётности и разрешения споров.

**XP при переводе в другую школу — сохраняется 50%** от накопленного значения, уровень пересчитывается по новой сумме.

**Для учителя** — история (что брал, когда) **переезжает вместе с ним** в новую школу (у учителя нет геймификации, поэтому вопрос "переносить ли рейтинг" не возникает — переносится просто лог).

## 8. Каталог: учебники vs обычные книги

### Учебники (Textbook + TextbookStock)
- `Textbook` — общий справочник для всех школ Узбекистана (один и тот же учебник по классу+предмету+учебному году), включает обложку (одна на класс+предмет).
- `TextbookStock` — школьный остаток экземпляров конкретного учебника (school + textbook + total/available copies).
- Выдача/сдача — **полностью ручная**, инициируется админом в любое время (не привязана жёстко к 1 сентября/маю как системная автооперация — массовая раздача 1 сентября — это просто применение той же ручной функции разом ко всем ученикам класса).
- Due date вычисляется в момент выдачи (обычно до конца текущего учебного года).
- Проверка: у ученика не может быть двух активных невозвращённых учебников по одному предмету одновременно.

### Обычные книги (RegularBook)
- Принадлежат конкретной школе, разные у всех школ.
- Одна обложка на всё название книги, независимо от количества экземпляров.
- Выдача/возврат — **исключительно через QR-код**, самообслуживание (ученик и учитель).
- Не ограничено количество единовременно на руках ("могут брать сколько угодно").

### UX выдачи учебников — "комплект по классу"
- У класса есть предзаполненный список предметов→учебников.
- Один экран "Выдать учебники ученику X" показывает весь комплект с чекбоксами и статусом по каждому предмету (не выдан/в наличии, не выдан/нет в наличии, уже выдан, просрочен).
- Аналогичный экран "Принять учебники от ученика X" для сдачи.
- Массовая версия — "выдать комплекты всем ученикам класса" одной операцией.

### Учитель и учебники
- Учитель берёт учебник **сам, через кнопку**, без участия админа — краткосрочный цикл (взял на урок → вернул после), технически ближе к модели обычной книги, чем к годовой выдаче ученику.
- По умолчанию видит только учебники своего предмета (5–11 класс), может снять фильтр при замене другого учителя (это фильтр интерфейса, НЕ ограничение прав доступа).
- Учитель может выбрать несколько классов/учебников и оформить одним QR-сканом разом (аналогично ученику с обычными книгами — см. корзину ниже).

## 9. QR/HMAC и процесс "корзины"

### HMAC-токен
- Payload включает: `student_id` (или `teacher_id`), `school_id`, timestamp, **список выбранных единиц** (книг/учебников).
- TTL — 2 минуты.
- Подпись — HMAC-SHA256, секрет на сервере.
- При валидации проверяется: подпись корректна, не истёк TTL, и **`school_id` в токене совпадает со школой сканирующего библиотекаря** — это исключает использование QR из одной школы в другой.

### Процесс "корзина"
1. Пользователь (ученик/учитель) ищет книги/учебники в каталоге, добавляет в промежуточный список ("корзина").
2. Подтверждает бронирование — система генерирует один QR-токен на всю группу.
3. Библиотекарь сканирует один раз — все позиции выдаются одной операцией.
4. **Возврат — тоже через QR, но выборочный**: пользователь заходит в "у меня на руках", отмечает, что возвращает сейчас (может быть часть, не всё), система генерирует QR на эту подгруппу, библиотекарь сканирует — только отмеченные позиции переходят в "возвращено", остальное остаётся выданным.
5. Каждая единица внутри группы имеет независимый статус — совместная выдача не означает совместный возврат.

### Аварийный/принудительный возврат (без QR)
- Только когда самообслуживание физически невозможно (нет доступа к аккаунту/интернету).
- Админ заходит в профиль пользователя → видит, что на руках → выбирает позицию → явное предупреждение-подтверждение ("вы уверены, что хотите принудительно принять без QR?") → подтверждает.
- Помечается отдельным статусом ("принудительный возврат"/"экстренный случай"), отдельно от обычного QR-возврата — для прозрачности, аналитики и защиты от злоупотреблений.

## 10. Новости (News) — видимость по иерархии

- `News.author_level` — `superadmin` или `school_admin`.
- Новость суперадмина: `school=null`, видна суперадмину и **всем школьным админам**, НЕ видна ученикам/учителям.
- Новость школьного админа: `school=<конкретная>`, видна ученикам/учителям **этой** школы + этому админу, НЕ видна другим школам.
- Школьный админ видит: новости суперадмина + свои собственные.
- Логика видимости инкапсулирована в `NewsService.visible_to(user)`.

## 11. Геймификация

### XP
- Возврат книги/учебника **вовремя** — полное начисление XP.
- Возврат **с опозданием** — половина от базового XP (щадящий штраф, не обнуление).
- **"Камбэк"-бонус** — если следующая после просрочки книга возвращена вовремя, начисляется дополнительный бонус сверху обычного XP ("молодец, вернулся в норму") — разовый, не путать со стриком.
- С ростом уровня — небольшой прогрессивный бонус к базовому начислению XP (конкретный процент/формула — параметр, легко настраиваемый).

### Уровни
- Прогрессивная шкала (больше XP для следующего уровня по мере роста) — точная формула фиксируется как конфигурационный параметр.

### Стрики
- По аналогии с популярными приложениями (тип Duolingo) — считается по дням активности, прерывается при пропуске дня; может быть механизм "заморозки" стрика.
- Достижения за вехи стрика: 1 неделя, 1 месяц, и т.д. — прогрессивная лестница.

### Достижения ("Зал достижений")
- Лёгкие визуальные элементы: **emoji + иконки из открытых библиотек (Lucide/Font Awesome) + CSS 3D-эффекты** (тени, градиенты, transform) — НЕ хранение тяжёлых PNG/растровых картинок на сервере, чтобы не раздувать ограниченный/бесплатный хостинг.
- Опционально — лёгкая 3D-визуализация на фронтенде (three.js), рендерится в браузере пользователя, не нагружает сервер.
- Ученик в своём профиле видит полный зал: открытые (чётко) + не открытые (приглушённо, с подсказкой условия).
- Для всех остальных (лидерборд, публичный профиль) — видны только уже полученные достижения, без "витрины" незавершённого.

### Лидерборды
- **Школьный** — по XP, все классы 5–11 вместе; видят ученики, учителя (без права участия), школьный админ.
- **По классу** — для классного руководителя (только его прикреплённые классы, read-only).
- **По школам (у суперадмина)** — по **средней** метрике на ученика (не по абсолютной сумме) — для честного сравнения школ разного размера. При клике на школу — краткая версия внутришкольного лидерборда.
- Учителя вообще не участвуют в геймификации, только наблюдают (обычный учитель — общешкольный рейтинг; классный руководитель — рейтинг своего класса).

## 12. Еженедельные челленджи (тесты)

- Раз в неделю, автоматически (Celery periodic task), для каждого класса 5–11 генерируется тест через внешний ИИ API (напр. Gemini, бесплатный тариф — нагрузка минимальна: ~11 запросов/неделю на всю систему).
- Вопросы — **простые, общей эрудиции**, под уровень возраста класса (финальная формулировка — простые общие вопросы, не углублённая предметная сложность).
- Формат: **15 вопросов, 3 варианта ответа**, показываются в **случайном порядке** (порядок фиксируется один раз при старте попытки и сохраняется в `ChallengeAttempt`, не пересчитывается при перезагрузке).
- **Общий таймер — 15 минут на весь тест** (не по 30 секунд на вопрос).
- Прогресс хранится на сервере (`ChallengeAttempt`: время начала, порядок вопросов, отвеченное) — переживает перезагрузку страницы; каждый ответ отправляется на сервер сразу, не копится локально.
- По истечении 15 минут — неотвеченные вопросы засчитываются как неверные/без XP, тест завершается автоматически.
- Правильные ответы начисляют XP; отдельный лидерборд по челленджам — **по классу/параллели отдельно** (не общешкольный, так как вопросы и сложность зависят от класса).
- Опциональный шаг модерации — школьный админ/учитель может просмотреть/поправить сгенерированные вопросы перед публикацией.
- Обработка ошибок API (недоступность/лимит/неожиданный формат ответа) — повтор попытки или уведомление админу, без "тихого" отсутствия вопросов у класса.

## 13. Excel-экспорты

Общий принцип: **пароль нигде не хранится в открытом виде и не показывается повторно** — только автоматический экспорт файла сразу после создания/сброса.

| Событие | Экспортируемые колонки |
|---|---|
| Создание школы | Школа, Район, ФИО админа, Логин, Пароль |
| Сброс пароля школьного админа | Школа, Логин, новый Пароль |
| Массовый импорт учеников/учителей (CSV) | ФИО, Класс, Логин, Пароль |
| Активация доступа 5 класса | ФИО, Класс, Логин, Пароль |
| Сброс пароля ученику/учителю | ФИО, новый Пароль |
| **Список к сдаче учебников (май)** | ФИО \| Кол-во учебников \| [Предмет 1] \| [Предмет 2] \| ... — колонки-предметы динамические, по комплекту класса. Каждая ячейка предмета: "Сдал"/"Не сдал" (перевод по **языку класса**, не по языку интерфейса админа — русский класс получает русскую таблицу, даже если у админа интерфейс на каракалпакском). Есть выбор при скачивании: **полный список** или **только должники**. |

Шаблон CSV для массового импорта — отдельная кнопка "Скачать шаблон" (колонки ФИО | Класс, без логина/пароля — генерируются системой).

## 14. Статистика и инвентаризация

- **Суперадмин:** активность районов, активность школ, рейтинг школ (по среднему), динамика во времени. Графики — Chart.js.
- **Школьный админ:** внутренняя инвентаризация — сколько книг/учебников всего и в наличии, сколько учеников активно читают, статистика по классам. Полноценный операционный дашборд.
- **1–4 класс** — фигурирует только в отчётности по учебникам (сколько выдано/в наличии), не в остальной статистике/рейтингах.

## 15. Frontend требования

- **Адаптивность** — единая вёрстка с breakpoints под desktop/tablet/mobile (не три отдельные версии сайта).
- **Near-realtime обновления** — HTMX polling (без перезагрузки страницы для отражения изменений); Django Channels/WebSockets — опционально для отдельных экранов в будущем, если потребуется настоящий realtime (не сразу, чтобы не нагружать ограниченный хостинг).
- **Страницы ошибок** — кастомные шаблоны 404/403/500 и т.д., оформленные в общем стиле, на языке пользователя (не дефолтные Django-страницы).
- **Тёмная/светлая тема** — тёмная тема контрастная, близкая к чёрной (не смягчённая синева) — учитывая, что это операционная CRM-панель с таблицами/данными, а не читалка сплошного текста.
- **Цветовая палитра:** основная — сине-зелёные/бирюзовые тона (образование, доверие, нейтрально для взрослых и детей); акцент для геймификации/XP/наград — тёплый золотой/оранжевый.

## 16. Явно исключено из v2 (осознанные решения)

- Никакого ИИ в рантайме на постоянной основе (риск лимитов бесплатных тарифов) — единственное исключение: генерация еженедельных челленджей (разовая фоновая операция, минимальная нагрузка).
- Нет автоперевода контента между языками — контент вводится вручную на нужном языке (привязанном к языку класса/школы).
- Нет email/SMS-верификации и восстановления пароля — только офлайн через админа.
- 1–4 класс не участвует в цифровом самообслуживании, геймификации, рейтингах — только учёт учебников.
- Внешний API для автоподгрузки данных о книгах (по ISBN) — отложено на будущее.

## 17. Открытые вопросы / детали для доработки на этапе реализации

- Точная формула прогрессии уровней (сколько XP на каждый уровень).
- Точный процент XP-бонуса за уровень и величина "камбэк"-бонуса.
- Условие "разрыва" стрика — ровно один пропущенный день, или есть льготный период/заморозки.
- Список конкретных достижений (полный перечень условий получения).
- Нужен ли похожий шаблон массового импорта для учителей отдельно от учеников.
- Формат/степень строгости модерации сгенерированных ИИ вопросов челленджа.
